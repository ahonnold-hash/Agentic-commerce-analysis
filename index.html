Cardholder Initiates → Agent Initiates

In the human model, a cardholder consciously initiates a transaction — applying their own judgment about value, trust, and affordability before touching the card. In an agent transaction, that conscious judgment is replaced by a programmatic decision. The question shifts from "does the cardholder want this?" to "is the agent authorized to want this on behalf of the cardholder?"

Network status: Partially addressed — with an important rules-verified nuance. Visa's rules (§4.1.24.3) require verifying the cardholder's identity before storing credentials — not the AI agent's identity. TAP verifies agent identity but exists at the product layer above the rules. This distinction matters: the rules confirm cardholder authorization, not agent authentication. Mastercard's Agentic Tokens scope agent authority to a consent policy. Amex's ACE captures purchase intent. All three address cardholder authorization; agent-level authentication is a product-layer addition, not a rules requirement for Visa.

⚠ The Authorization Gap — The Missing Issuer

In a card transaction, authorization can fail at two distinct points: rejected by the network (missing mandatory data fields — BIN, MCC, amount, currency) or declined by the issuer (fraud signals, velocity limits, risk criteria). The agent equivalent of both failure modes exists — but the intelligence to trigger the value dimension is entirely missing. Networks reject malformed payment data. Nobody rejects an agent transacting at an unjustified price.

The Two Failure Modes — Both Unsolved for Value

Failure Mode 1 — Network Rejection: Missing mandatory fields → transaction rejected before reaching issuer. For value: a model scoring near zero (like Buffer) should be rejected at the agent evaluation gate — "dead on arrival," no partial credit.

Failure Mode 2 — Issuer Decline: Request reaches issuer but fails risk/fraud criteria. For value: a model that passes structural parsing but fails EVE evaluation — reaches the gate but the agent can't confirm value justifies price.

Cardholder Benefits → Value Realization Verification

In the human model, benefit is experienced by the cardholder based on their own post-purchase judgment. In an agent transaction, the system needs to compute whether the transaction delivered the value the cardholder intended — ideally before executing, and verifiably after. None of the three networks provide this computation. Only The Value Project addresses it through EVE models linked to billing and usage data.

Rules-verified finding: Under Visa Core Rules §4.1.24.10, the cardholder bears full liability for agent actions — the same as if they personally initiated the transaction. Zero Liability (§1.4.5.1) only covers unauthorized transactions. A properly-consented agent purchase that delivers poor value or pays an unjustified price is authorized — and the cardholder has no Visa-backed protection. This is the exact inverse of Amex's Agent Purchase Protection, which covers erroneous agent transactions. The value intelligence layer isn't just a market opportunity — it's the consumer protection backstop the Visa rulebook currently lacks.

▼ 04 The Intelligence Gap — What No Network Has Built Value Project opportunity Critical

The Core Finding

All three networks have invested heavily in Layer 4 — payment execution. None have addressed Layers 1–3 or 5 — the intelligence layers above the payment rail. Visa's ICC catalog discoverability feature is the single closest move any network has made toward the intelligence layer. It is insufficient on its own. The Value Project is the only open standard addressing this gap.

Gap 01
Catalog ≠ Value Model
Visa ICC gets merchant product data onto AI platforms — descriptions, specs, prices. An agent can see a product costs $200. It cannot compute whether $200 is justified, what the economic value delivered is, or how it compares to alternatives. Value Project EVE schemas bridge this gap directly.
Gap 02
Credential Scope ≠ Value Authorization
Mastercard's Agentic Token scopes an agent to a specific merchant and consent policy. It solves "is this agent allowed to buy here?" It does not solve "is the price this merchant is charging justified by economic value delivered?" The Agentic Token and a Value Project pricing schema are complementary — not competing.
Gap 03
Protection ≠ Value Realization
Amex's Agent Purchase Protection is the industry's most honest acknowledgment that agents will make mistakes. Protection after a bad transaction is not value verification before it. Value Project schemas linked to billing/usage data turn Amex's protection layer from reactive backstop into proactive verification — a partnership waiting to be built.
Gap 04
Two-Tier Failure Model Unaddressed
In payments, authorization fails at the network level (malformed data) or issuer level (risk criteria). Value models need both: a minimum viable data threshold (below which an agent simply cannot parse the model) and a value confidence threshold (below which an agent should not transact with confidence). No network has defined either threshold. The Value Project rubrics are the only framework addressing this.
Gap 05 — Rules-Verified
⚠ Liability Asymmetry: Visa vs. Amex
Visa Core Rules §4.1.24.10 place full liability for agent errors on the cardholder. Zero Liability (§1.4.5.1) only covers unauthorized transactions — a properly-consented bad-value agent purchase is authorized and unprotected. Amex's Agent Purchase Protection is the exact opposite: erroneous agent transactions are explicitly covered. This liability asymmetry is the most consequential enterprise merchant consideration in the rulebook. Merchants on Visa-only rails whose agents make bad-value purchases expose their customers with no network backstop. The Value Project's pre-transaction value verification layer is — under the Visa rulebook — the only mechanism available to prevent this exposure.
Gap 06 — Rules-Verified
TAP and ICC Are Not in the Rulebook
Neither "Trusted Agent Protocol" nor "Intelligent Commerce Connect" appears anywhere in Visa's 923-page Core Rules (April 18, 2026). They are product-layer constructs — marketing names for capabilities built above the rules framework. The only rules-mandated Visa agentic construct is enrollment in the Visa Intelligent Commerce program (§4.1.24.1–2). Enterprise merchants should understand that TAP and ICC can be modified or discontinued by Visa without rules changes — they carry the governance weight of product features, not network rules.
